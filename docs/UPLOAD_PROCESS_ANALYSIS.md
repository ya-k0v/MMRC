# Анализ процесса загрузки файлов

## Общая схема процесса

```
Клиент → POST /api/devices/:id/upload
  ↓
[Multer] Сохранение файла в /content/ (shared storage)
  ↓
[files.js] Обработка загруженных файлов
  ↓
[file-metadata-processor.js] Обработка метаданных
  ├─ ШАГ 1: Faststart обработка MP4 (синхронно)
  ├─ ШАГ 2: Вычисление MD5 обработанного файла
  ├─ ШАГ 3: Проверка дедупликации
  ├─ ШАГ 4: Сохранение метаданных в БД
  └─ ШАГ 5: Генерация трейлера (фоновая)
  ↓
[optimizer.js] Автоматическая оптимизация видео (фоновая)
```

## Детальный разбор этапов

### 1. Получение запроса (POST /api/devices/:id/upload)

**Файл:** `src/routes/files.js:535`

**Middleware:**
- `uploadLimiter` - ограничение частоты запросов
- `upload.array('files', 50)` - Multer middleware для загрузки до 50 файлов

**Валидация:**
- Проверка device_id
- Проверка существования устройства
- Проверка расширений файлов (ALLOWED_EXT)

### 2. Сохранение файла (Multer)

**Файл:** `src/middleware/multer-config.js`

**Процесс:**
1. **Destination:** `/content/` (shared storage через `getDevicesPath()`)
2. **Filename:** 
   - Транслитерация оригинального имени → безопасное имя
   - Если файл существует → добавляется суффикс `_xxx`
   - Сохранение маппинга оригинального → безопасное имя в `req.originalFileNames`

**Результат:**
- Файл сохранен в `/content/{safeName}`
- Оригинальное имя сохранено в маппинге

### 3. Обработка специальных файлов

**Файл:** `src/routes/files.js:575-927`

**PDF/PPTX/ZIP:**
- Перемещаются в `/content/{device}/` (не в shared storage)
- Автоматическая конвертация PDF/PPTX в изображения

**Папки (folderName):**
- Создается папка `/content/{device}/{folderName}/`
- Файлы перемещаются в папку

### 4. Обработка метаданных (file-metadata-processor.js)

**Файл:** `src/utils/file-metadata-processor.js:23`

**ПОРЯДОК ОПЕРАЦИЙ (КРИТИЧНО!):**

#### ШАГ 1: Faststart обработка MP4 (строки 47-86)
- **Синхронно** (await) - ждем завершения
- Выполняется ДО вычисления MD5
- Использует `applyFaststart(filePath, { checkFirst: false })`
- Команда: `ffmpeg -i file -c copy -movflags +faststart output`
- **Результат:** moov атом перемещен в начало файла

#### ШАГ 2: Вычисление MD5 (строки 88-101)
- Вычисляется для **обработанного** файла
- Для больших файлов (>100MB): partial MD5 (первые 10MB) + full MD5
- Для маленьких: только full MD5

#### ШАГ 3: Дедупликация (строки 103-150)
- Проверка дубликатов по MD5 обработанного файла
- Если найден дубликат:
  - Удаляется обработанный новый файл
  - Используется существующий файл из shared storage
  - Копируются метаданные из источника

#### ШАГ 4: Извлечение метаданных (строки 152-200)
- Если дедуплицирован → копируются метаданные из источника
- Если новый файл → извлечение через FFmpeg:
  - Видео: width, height, duration, codec, profile, bitrate
  - Аудио: codec, bitrate, channels

#### ШАГ 5: Сохранение в БД (строки 202-240)
- Сохранение всех метаданных в `files_metadata` таблицу
- file_path указывает на shared storage или device folder

#### ШАГ 6: Фоновая обработка (строки 242-270)
- Генерация трейлера (5 секунд) - фоновая
- Faststart для дедуплицированных файлов - фоновая

### 5. Автоматическая оптимизация видео

**Файл:** `src/routes/files.js:906-927`

**Процесс:**
- Запускается **после** сохранения метаданных
- Проверяет нужна ли оптимизация (разрешение, профиль, битрейт)
- Если нужна → запускает перекодировку в фоне
- Использует `autoOptimizeVideo()` из `src/video/optimizer.js`

## Критические моменты

### ✅ Правильный порядок операций

1. **Faststart ДО MD5** - важно!
   - MD5 вычисляется для обработанного файла
   - Дедупликация работает с обработанными файлами
   - Все файлы гарантированно обработаны

2. **Синхронная обработка MP4**
   - `await applyFaststart()` - ждем завершения
   - Гарантирует что файл обработан перед дедупликацией

3. **Дедупликация после обработки**
   - Экономия места сохраняется
   - Обработанные файлы дедуплицируются правильно

### ⚠️ Потенциальные проблемы

1. **Время обработки**
   - Faststart обработка синхронная - может замедлить ответ
   - Для больших файлов может занять несколько секунд
   - **Решение:** Обработка быстрая (copy, не перекодирование)

2. **Ошибки обработки**
   - Если faststart падает - файл все равно сохраняется
   - Продолжаем обработку даже при ошибке (строки 78-85)

3. **Дедупликация обработанных файлов**
   - Если обработанный файл дедуплицирован - используется существующий
   - Существующий файл может быть не обработан
   - **Решение:** Проверка faststart для дедуплицированных файлов в фоне (строки 245-270)

## Потоки данных

### Обычный файл (не дедуплицирован):
```
Upload → /content/file.mp4
  ↓
Faststart обработка (синхронно)
  ↓
MD5 вычисление
  ↓
Сохранение метаданных в БД
  ↓
file_path = /content/file.mp4
```

### Дедуплицированный файл:
```
Upload → /content/file.mp4
  ↓
Faststart обработка (синхронно)
  ↓
MD5 вычисление
  ↓
Найден дубликат → удаление /content/file.mp4
  ↓
Сохранение метаданных в БД
  ↓
file_path = /content/existing_file.mp4 (существующий)
  ↓
Проверка faststart для существующего файла (фоновая)
```

## Производительность

### Время обработки (примерно):
- **Faststart (copy):** 1-5 секунд для файла 100MB
- **MD5 вычисление:** 1-3 секунды для файла 100MB
- **Дедупликация:** <100ms (поиск в БД)
- **Сохранение метаданных:** <50ms

### Общее время:
- **Маленький файл (<50MB):** ~2-5 секунд
- **Большой файл (>100MB):** ~5-15 секунд

## Рекомендации

1. ✅ **Текущий порядок правильный** - faststart перед MD5
2. ✅ **Синхронная обработка** - гарантирует корректность
3. ⚠️ **Мониторинг времени** - для больших файлов может быть медленно
4. ✅ **Обработка дедуплицированных** - фоновая проверка работает
